================================================================
  RECALLAI — STEP BY STEP BUILD PLAN
================================================================

A RAG-powered student note app built with LangChain, LangGraph,
FastAPI backend + React Vite with shadcn/ui frontend.

Students organize notes in a workspace → class → folder → file
hierarchy. Files and text notes are uploaded independently —
FastAPI returns immediately and parsing/indexing happens in a
background task. All vectors are stored in Amazon OpenSearch.
All metadata is stored in PostgreSQL (RDS).

  Workspace (the student)
  └── Class: Biology 101
  │   ├── Folder: Midterm
  │   │   ├── lecture_3.pdf
  │   │   └── my_notes.txt
  │   └── Folder: Final
  │       └── chapter_summary.pdf
  └── Class: History
      └── Folder: Essay Research
          └── sources.pdf

  Upload pipeline (file):
    Browser → POST /api/files/upload (multipart)
    FastAPI → save to S3 → create DB record (status=processing)
    FastAPI → return {file_id, status} immediately
    BackgroundTask → parse → chunk → embed → OpenSearch
    BackgroundTask → update status=ready in PostgreSQL
    Frontend polls GET /api/files/{id}/status every 2s

  Upload pipeline (text note):
    Browser → POST /api/notes/text
    FastAPI → save raw text to S3 as .txt → create DB record
    FastAPI → return {note_id, status} immediately
    BackgroundTask → chunk → embed → OpenSearch
    BackgroundTask → update status=ready in PostgreSQL

  Both pipelines share the same parser/chunker/embedder code.
  Only the trigger changes — multipart upload vs text body.

4 phases, ~8 weeks to a working product.

Step 4 — Build the parsing pipeline (pipeline/)

  All three modules are shared by both the file upload and
  text note routes — only the entry point differs.

  parser.py — extract raw text by input type:
    .pdf   → PyMuPDF — text + page number per page
    .pptx  → python-pptx — text per slide
    .docx  → python-docx — paragraphs + tables
    .txt   → read directly
    text   → already a string, pass through as page 1
    Returns: list of {text, page_or_slide}

  chunker.py — split into chunks:
    RecursiveCharacterTextSplitter
    chunk_size=500, chunk_overlap=50
    Preserve page/slide number in each chunk metadata

  embedder.py — embed and store:
    Embed with all-MiniLM-L6-v2 (384 dimensions)
    Bulk index into OpenSearch "notes" with full metadata:
      content, embedding, file_id, file_name,
      class_id, class_name, folder_id, folder_name, page

Step 5 — Build the file upload route (routes/files.py)

  FastAPI receives the file, saves it to S3, creates a DB
  record, fires a background task, and returns immediately.

  @router.post("/folders/{folder_id}/upload")
  async def upload_file(
      folder_id: str,
      file: UploadFile,
      background_tasks: BackgroundTasks,
      db: AsyncSession = Depends(get_db)
  ):
      # 1. Upload to S3
      s3_key = f"workspaces/.../folders/{folder_id}/{file.filename}"
      await s3.upload_fileobj(file.file, S3_BUCKET, s3_key)

      # 2. Create file record in PostgreSQL (status=processing)
      file_record = await db.create_file(
          name=file.filename,
          s3_key=s3_key,
          folder_id=folder_id,
          status="processing"
      )

      # 3. Fire background task — returns to client immediately
      background_tasks.add_task(
          run_pipeline,
          file_id=file_record.id,
          s3_key=s3_key,
          file_type=file.content_type,
          metadata=file_record.metadata
      )

      return {"file_id": file_record.id, "status": "processing"}

  run_pipeline (the background task):
    async def run_pipeline(file_id, s3_key, file_type, metadata):
        try:
            raw = await s3.download(s3_key)
            pages = parser.parse(raw, file_type)
            chunks = chunker.split(pages)
            await embedder.index(chunks, metadata)
            await db.update_file_status(file_id, "ready")
        except Exception as e:
            await db.update_file_status(file_id, "error", str(e))

  File status endpoint (frontend polls this):
    GET /api/files/{id}/status
    → SELECT status, error_reason FROM files WHERE id = ?
    → returns {file_id, status, error_reason}

  Delete endpoint (removes from S3 + OpenSearch + PostgreSQL):
    DELETE /api/files/{id}
    → delete S3 object
    → delete OpenSearch chunks where file_id = ?
    → delete file row from PostgreSQL

Step 6 — Build the text note route (routes/notes.py)

  Text input follows the exact same pipeline. FastAPI writes
  the text to S3 as a .txt file so storage is consistent,
  then fires the same background task.

  @router.post("/folders/{folder_id}/notes")
  async def create_text_note(
      folder_id: str,
      body: TextNoteInput,   # {title: str, content: str}
      background_tasks: BackgroundTasks,
      db: AsyncSession = Depends(get_db)
  ):
      # 1. Save text to S3 as a .txt file
      note_id = str(uuid4())
      s3_key = f"workspaces/.../folders/{folder_id}/{note_id}.txt"
      await s3.put_object(Bucket=S3_BUCKET, Key=s3_key,
                          Body=body.content.encode())

      # 2. Create file record (file_type=text_note, status=processing)
      file_record = await db.create_file(
          name=body.title,
          s3_key=s3_key,
          folder_id=folder_id,
          file_type="text_note",
          status="processing"
      )

      # 3. Same background task as file upload
      background_tasks.add_task(
          run_pipeline,
          file_id=file_record.id,
          s3_key=s3_key,
          file_type="text_note",
          metadata=file_record.metadata
      )

      return {"file_id": file_record.id, "status": "processing"}

  Both routes call the same run_pipeline() function.
  The parser handles file_type="text_note" by reading the
  S3 content directly as plain text — no special parsing needed.

Step 7 — Build qa.py with scoped OpenSearch retrieval

  Scope levels:
    {"class_id":  "bio101"}   → whole class
    {"folder_id": "midterm"}  → single folder
    {"file_id":   "abc123"}   → single file
    {}                        → search everything

  OpenSearch scoped KNN query:
    {
      "size": 4,
      "query": {
        "bool": {
          "must": {
            "knn": { "embedding": { "vector": [...], "k": 4 } }
          },
          "filter": [
            { "term": { "class_id": "bio101" } }
          ]
        }
      }
    }

  Return: answer + source metadata (file name, page, folder)

================================================================
  HOW THE TWO UPLOAD FLOWS WORK SIDE BY SIDE
================================================================

  FILE UPLOAD:
    Browser → POST /api/folders/{id}/upload (multipart)
    FastAPI → boto3 upload to S3
    FastAPI → INSERT file row (status=processing) → PostgreSQL
    FastAPI → background_tasks.add_task(run_pipeline, ...)
    FastAPI → return {file_id, status} ← client gets this instantly
    BackgroundTask → parser.py (PDF/PPTX/DOCX → pages)
    BackgroundTask → chunker.py → embedder.py → OpenSearch
    BackgroundTask → UPDATE status=ready → PostgreSQL
    Frontend polls GET /api/files/{id}/status every 2s

  TEXT NOTE:
    Browser → POST /api/folders/{id}/notes {title, content}
    FastAPI → boto3 put_object (text as .txt to S3)
    FastAPI → INSERT file row (file_type=text_note, status=processing)
    FastAPI → background_tasks.add_task(run_pipeline, ...)
    FastAPI → return {file_id, status} ← client gets this instantly
    BackgroundTask → parser.py (reads .txt → single page)
    BackgroundTask → chunker.py → embedder.py → OpenSearch
    BackgroundTask → UPDATE status=ready → PostgreSQL
    Frontend polls GET /api/files/{id}/status every 2s

  run_pipeline() is called identically in both cases.
  file_type tells parser.py which extraction path to take.

  Week 1:  Python async/await, FastAPI BackgroundTasks
           SQLAlchemy async + asyncpg, Alembic migrations
           PostgreSQL — UUID keys, JSONB, indexes

  Week 2:  Embeddings + cosine similarity
           OpenSearch — KNN index mapping, filter queries
           boto3 — S3 upload, download, put_object

  Week 3:  LangChain — loaders, splitters, retrievers, memory
           File parsing — PyMuPDF, python-pptx, python-docx
           OpenSearchVectorSearch LangChain integration

  Week 4:  React + Vite + shadcn, recursive tree component
           Polling pattern for async status updates
           Fetch + SSE streaming

  Week 5:  LangGraph — state with scope, nodes, conditional edges

  Week 6:  PostgreSQL analytics queries (GROUP BY, JSONB ops)

  Week 7+: RDS + RDS Proxy, ECS deployment,
           AWS Secrets Manager, production hardening


================================================================
  GOLDEN RULE
================================================================

  Build a working thing first, then improve it.

  For weeks 1-2: skip AWS entirely.
    - Run PostgreSQL locally with Docker
    - Run OpenSearch locally with Docker
    - Use the local filesystem instead of S3
      (swap boto3 calls for open()/write() during dev)
    - BackgroundTasks work identically locally and in production

  When ready for AWS: swap the local filesystem helpers in s3.py
  for real boto3 calls. Everything else stays the same.

  Local dev commands:
    docker run -p 5432:5432 -e POSTGRES_PASSWORD=secret postgres:16
    docker run -p 9200:9200 opensearchproject/opensearch:2
